The SM83 Core

The Game Boy's processor is often called a "Z80" — it is not. It is a Sharp SM83 (sometimes labelled LR35902), a custom 8-bit CPU that borrows from both the Intel 8080 and the Zilog Z80 but is neither. It runs at 4.194304 MHz (2^22 Hz, derived from a 4 MiHz crystal). The system clock is divided internally: the CPU executes at 1.048576 MHz (one machine cycle = 4 T-cycles), meaning every opcode takes a minimum of 4 clock ticks.

The register file is pure 8080 heritage: eight 8-bit registers A, F, B, C, D, E, H, L, where A is the accumulator and F is the flags register (bits 7–4: Zero, Negative (subtract), Half-carry, Carry; bits 3–0 are always zero). The 8-bit registers pair into 16-bit pseudo-registers — AF, BC, DE, HL — for 16-bit addressing. Two dedicated 16-bit registers complete the set: SP (stack pointer, initialised to 0xFFFE) and PC (program counter, starting at 0x0100 after the boot ROM hands off).

From the Z80, the SM83 inherits the CB-prefixed rotation, shift, and bit-test instructions — a second 256-entry opcode table accessed via a 0xCB prefix byte. But it drops the Z80's IX and IY index registers, the shadow register bank (AF', BC', DE', HL'), the I/O port space instructions (IN/OUT), and most of the ED-prefixed block. In their place, it adds unique instructions: STOP (halts the CPU and LCD until a button press), SWAP (exchanges the high and low nibbles of a register), and modified LDH instructions for fast access to the 0xFF00–0xFFFF I/O page. There is no instruction pipeline, no cache, no prefetch — every instruction is fetched, decoded, and executed sequentially.

The Address Map

The SM83 has a 16-bit address bus, giving it a 64 KB address space — every byte from 0x0000 to 0xFFFF is mapped to a specific hardware resource. There is no MMU, no virtual memory, no protection rings. A read from address 0x8000 always hits VRAM; a write to 0xFF40 always hits the LCD control register. The entire system is memory-mapped I/O.

The first 32 KB (0x0000–0x7FFF) is mapped to the cartridge ROM. The lower 16 KB (0x0000–0x3FFF) is fixed to bank 0; the upper 16 KB (0x4000–0x7FFF) is switchable via the cartridge's Memory Bank Controller (MBC). An MBC1 cartridge, for example, supports up to 128 ROM banks (2 MB total), selectable by writing the desired bank number to the address range 0x2000–0x3FFF — a write to ROM that the MBC intercepts. This is the Game Boy's answer to a 16-bit address bus needing to access more than 64 KB of data.

VRAM occupies 0x8000–0x9FFF (8 KB): the first 6 KB holds tile pixel data (up to 384 tiles), and the last 2 KB holds two 32×32 tile maps. External RAM — the battery-backed SRAM on the cartridge — is mapped to 0xA000–0xBFFF (8 KB, also bankable). Work RAM (WRAM) fills 0xC000–0xDFFF (8 KB), with 0xE000–0xFDFF being a mirror (echo RAM) due to incomplete address decoding. The OAM (Object Attribute Memory) at 0xFE00–0xFE9F holds the 40 sprite entries (4 bytes each: Y, X, Tile Index, Attributes). The I/O registers (0xFF00–0xFF7F) control every peripheral. HRAM (0xFF80–0xFFFE), the 127-byte high-speed memory accessible during DMA, and the IE register at 0xFFFF complete the map.

The Pixel Pipeline

The PPU (Pixel Processing Unit) drives a 160×144 pixel LCD at approximately 59.7275 Hz. It does not render frames — it renders scanlines. Each of the 154 scanlines in a frame (144 visible + 10 V-Blank) takes exactly 456 T-cycles (114 machine cycles), giving a total frame time of 70,224 T-cycles.

Each visible scanline passes through three modes in strict sequence. Mode 2 (OAM Scan) lasts 80 T-cycles: the PPU searches the 40 OAM entries to find which sprites overlap the current scanline (maximum 10 per line). Mode 3 (Pixel Transfer) takes 168–291 T-cycles (variable, depending on sprite count and window activity): the PPU reads tile data from VRAM and pushes pixels through an internal FIFO (First-In-First-Out) pipeline to the LCD driver. Mode 0 (H-Blank) fills the remainder of the 456-cycle line — this is the only time the CPU can safely access VRAM without bus conflicts.

After all 144 visible lines, the PPU enters Mode 1 (V-Blank) for 10 scanlines (4,560 T-cycles). The V-Blank interrupt fires, and the CPU typically uses this window to update VRAM, modify scroll registers, or swap tile data. The PPU's state machine is controlled by the LCDC register (0xFF40), which toggles the LCD, selects tile data addressing modes, enables the window and sprite layers, and sets sprite size (8×8 or 8×16). The STAT register (0xFF41) reports the current mode and drives the LCD STAT interrupt.

The Tile Engine

The Game Boy's PPU is a tile-based renderer. There are no bitmapped framebuffers — every pixel on screen is derived from an 8×8 tile. Each tile is stored in VRAM as 16 bytes: two bytes per row, using a 2 bits-per-pixel (2bpp) planar encoding. For each pixel in a row, the colour index is formed by taking one bit from the first byte (low bit) and one bit from the second byte (high bit), giving four possible values: 0, 1, 2, or 3.

These colour indices do not map directly to physical colours. Instead, they pass through a palette register: BGP (0xFF47) for background tiles, OBP0 (0xFF48) and OBP1 (0xFF49) for sprites. Each palette register is an 8-bit value encoding four 2-bit greyscale mappings. On the original DMG, colour 0 = lightest (#9BBC0F), colour 3 = darkest (#0F380F). This indirection is how the Game Boy achieves palette-swap effects without modifying tile data.

The background layer is a virtual 256×256 pixel plane (32×32 tiles), of which the 160×144 LCD viewport is a scrollable window controlled by SCX (0xFF43) and SCY (0xFF42). The tile map at 0x9800 or 0x9C00 (selected by LCDC bit 3) is a 32×32 array of tile indices pointing into the tile data region. The window layer is a second overlay controlled by WX (0xFF4B) and WY (0xFF4A), commonly used for HUD elements. Sprites (objects) are defined in OAM: each entry specifies an X/Y position, a tile index, and attribute flags (priority, X-flip, Y-flip, palette). The PPU composites background, window, and sprites per-pixel during Mode 3, resolving priority through the BG-over-OBJ bit and the OAM ordering rules.

The Sound Chip

The APU generates audio through four independent channels, each hardwired to a specific synthesis method. Channel 1 is a pulse wave with frequency sweep: it has a configurable duty cycle (12.5%, 25%, 50%, 75%), an 11-bit frequency register, a 4-bit volume envelope (with configurable up/down direction and period), and a sweep unit that can automatically slide the frequency up or down at a programmable rate — this is how classic Game Boy sound effects achieve pitch bends.

Channel 2 is identical to Channel 1 but without the sweep unit. Channel 3 is a wave channel: it plays back a 32-sample, 4-bit waveform stored in registers 0xFF30–0xFF3F (16 bytes, 2 samples per byte). By writing custom waveforms, programmers can approximate any periodic signal — triangle, sawtooth, or arbitrary shapes. Channel 4 is a noise channel driven by a 15-bit LFSR (Linear Feedback Shift Register) with an optional 7-bit mode for metallic tones. The LFSR's clock divisor and shift frequency are configurable, allowing everything from white noise to tuned percussion.

All four channels mix into a stereo output via the NR51 register (0xFF25), which routes each channel to the left speaker, right speaker, or both. Master volume for each side is controlled by NR50 (0xFF24). The APU runs on a frame sequencer clocked at 512 Hz (derived from the DIV timer), which steps the length counters (at 256 Hz), volume envelopes (at 64 Hz), and frequency sweep (at 128 Hz) in a fixed 8-step sequence.

Interrupts, Timers & DMA

The SM83 supports five maskable interrupts, each with a fixed vector address. V-Blank (0x0040) fires at the start of every V-Blank period. LCD STAT (0x0048) fires on configurable PPU events — mode transitions, LY==LYC coincidence. Timer (0x0050) fires when the TIMA register overflows. Serial (0x0058) fires after a serial transfer completes. Joypad (0x0060) fires on a button press (active-low transition). Interrupts are enabled globally by the IME flag (set by EI or RETI, cleared by DI) and individually by the IE register (0xFFFF). Pending interrupts are latched in the IF register (0xFF0F). When an interrupt fires, the CPU pushes the current PC onto the stack, clears IME, and jumps to the handler vector — a 20 T-cycle operation.

The timer system is driven by a 16-bit internal counter (the DIV counter) that increments at the full 4 MHz clock rate. The DIV register (0xFF04) exposes the upper 8 bits, effectively counting at 16,384 Hz. Writing any value to DIV resets the entire 16-bit counter to zero. The TIMA register (0xFF05) increments at a rate selected by TAC (0xFF07): 4,096 Hz, 262,144 Hz, 65,536 Hz, or 16,384 Hz. When TIMA overflows, it is reloaded from TMA (0xFF06) and the Timer interrupt is requested. Crucially, the TIMA increment is implemented by detecting a falling edge on a specific bit of the internal DIV counter — this quirk causes a "timer glitch" where writing to DIV can trigger an unexpected TIMA increment.

The OAM DMA transfer is the Game Boy's only DMA mechanism. Writing a source address (divided by 0x100) to the DMA register (0xFF46) initiates a 160-byte transfer from the source region to OAM (0xFE00–0xFE9F), taking 640 T-cycles (160 machine cycles). During this transfer, the CPU can only access HRAM (0xFF80–0xFFFE) — all other memory regions are locked by the DMA controller. This is why every Game Boy game contains a small DMA-trigger routine copied into HRAM at boot: the CPU must execute from HRAM while DMA runs.
